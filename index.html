第一周：搞懂"测试到底在干嘛"

第 1–7 天
1
软件测试是什么？
别急着学技术。先搞清楚：测试岗位到底干嘛？和开发什么区别？看 2 篇知乎/B站入门文章。关键词：软件测试是什么、测试日常工作。
⏱ 1.5 小时
2
黑盒 vs 白盒 vs 灰盒
功能测试属于"黑盒测试"——不看代码，只看功能对不对。理解三个概念，能用自己的话说出来就行。
⏱ 1.5 小时
黑盒白盒灰盒
3
等价类划分法（重点！）
面试和实际工作最常用的方法。把输入数据分"有效"和"无效"两组，每组挑代表值测试。比如年龄输入框：有效类=1~120，无效类=<0和>120。拿纸笔自己画几个例子。
⏱ 2 小时
等价类核心方法
4
边界值分析（重点！）
Bug 最爱藏在边界。比如输入限制 6-18 位密码：测 5 位、6 位、18 位、19 位。把昨天等价类的例子全部加边界值验证。面试官考这个超过 80%。
⏱ 2 小时
边界值必考
5
因果图 + 场景法
多个条件组合怎么测？画因果图穷举。用户操作流程怎么测？场景法从头走到尾。今天重点是"能看懂"，不用精通。
⏱ 1.5 小时
因果图场景法
6
写一份测试用例（动手！）
找一个你常用 App 的登录功能。用 Excel 写完整的测试用例表。列：编号、测试点、前置条件、操作步骤、预期结果。写到至少 15 条。
⏱ 2 小时
动手Excel
7
Bug 生命周期 + 写好 Bug 报告
Bug 怎么流转：新建→确认→修复→验证→关闭。好报告长什么样？标题一句话说清，复现步骤编号写清楚，截图或录屏。去 App 里实际找 3 个 Bug 写下来。
⏱ 2 小时
缺陷管理动手

第二周：学会工具，能干活了

第 8–14 天
8
Postman 入门：发第一个请求
安装 Postman → 新建 Request → 调一个公开 API（比如百度搜索建议接口）→ 看返回的 JSON。今天只要会 GET 请求、看 Status Code、看 Response Body。
⏱ 2 小时
Postman动手
9
Postman：POST 请求 + 参数
GET 会了，今天学 POST。构造带 JSON Body 的请求，设置 Headers（Content-Type）。理解 Query Params 和 Body 的区别。找公开 API 练习。
⏱ 2 小时
PostmanPOST
10
学会"看"接口返回
调接口不是目的，验证才是。200 不一定对，404 不一定是后端的问题。学会看 Status Code（200/400/401/403/404/500）、Response 结构对不对、数据值对不对。面试官想听这个。
⏱ 2 小时
验证思维核心
11
Charles 抓包入门
装好 Charles → 开启抓包 → 浏览器随便访问网站 → 看 HTTP 请求长什么样。关注：Request URL、Method、Headers、Response Status。能抓到包、看懂结构就过关。
⏱ 2 小时
Charles抓包
12
用 Charles 定位"前端 or 后端"问题
核心技能：前端说"接口返回错了"、后端说"我返回是对的"→ 开 Charles 抓包看 Request/Response，铁证如山。今天模拟一次：故意改请求参数看返回变化。
⏱ 1.5 小时
定位问题实战
13
MySQL：SELECT 和 WHERE
数据库不会整，功能测试做不下去。今天学 SELECT * FROM 表 WHERE 条件。用 sqlzoo.net 做前 5 道练习题。重点是 WHERE 的各种写法。
⏱ 2 小时
MySQL必学
14
MySQL：JOIN 联表查询
面试必考！搞懂 INNER JOIN。SELECT a.name, b.order_no FROM users a JOIN orders b ON a.id=b.user_id。在 sqlzoo 做完 JOIN 章节。
⏱ 2 小时
JOIN面试必考

第三周：完整走一遍测试流程

15

执行测试
拿出用例表一条一条执行。实际结果和预期不一样？→ Bug。每发现一个 Bug，用 Postman/Charles 辅助定位是前端还是后端的问题。

19
写 Bug 报告
把发现的 Bug 写成正式报告（Excel 或 Word）。每个 Bug：标题、环境、严重等级、复现步骤、预期结果、实际结果、截图。至少 5 个。

📋 面试高频 7 问

点击问题展开答案。建议先自己想一遍再看。
1给你一个登录页面，怎么设计测试用例？▾
分 5 个维度，每维度说 1-2 条：

UI：输入框对齐、占位文字、Tab 切换顺序、密码掩码。
功能：正确账号登录、错误密码、空账号、空密码、含空格。
边界：最大/最小长度±1、复制粘贴输入。
安全：密码明文传输？连续输错锁定？SQL 注入 ' OR '1'='1、XSS、退出后浏览器后退。
兼容：Chrome/Firefox/Edge、手机浏览器。

关键不是背，是分类有条理。
2你提了 Bug，开发说不是 Bug，怎么办？▾
5 步走：
1. 查需求文档——有写就拿出来，铁证。
2. 甩证据——截图/录屏/Charles 抓包，摆数据。
3. 走过去聊——别在系统里吵，当面一句话说清。
4. 拉产品——需求模糊就找产品经理确认。
5. 升级——邮件抄送主管，列证据+影响。

态度：对事不对人。目标是搞清楚这到底是不是缺陷。
3等价类和边界值有什么区别？举例。▾
等价类：把输入分组，每组值效果一样，只测代表值。
边界值：组和组之间的分界线最容易出 Bug，重点测。

举例——密码限 6-18 位：
等价类：有效类=6-18位（测10位），无效类=<6位和>18位。
边界值：5位(刚好不够)、6位(刚好够)、18位(最大)、19位(刚好超)。

一句话：等价类告诉你"测哪"，边界值告诉你"重点测哪"。
4缺陷的生命周期？▾
新建 → 确认 → 修复中 → 已修复 → 已验证 → 关闭

分支：开发拒绝→不予解决；验证不通过→重新打开；延期→暂缓。
关键：Bug 必须由提的人验证关闭，不能开发自己关。
5MySQL JOIN 怎么用？写个例子。▾
INNER JOIN（最常用）：只返回两表都匹配的行。
LEFT JOIN：左表全保留，右表没匹配的填 NULL。

写出来：
SELECT u.name, o.order_no
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE o.status = 'paid';

这句查出"已支付订单的用户名+订单号"。面试时口述思路比写对语法更重要。
6Postman 怎么测一个 POST 接口？▾
1. 新建 Request，方法选 POST
2. 填 URL
3. Headers 加 Content-Type: application/json
4. Body 选 raw → JSON，填数据
5. 点 Send
6. 验证：Status Code 对没对？Body 结构对不对？数据值对不对？

重点是"不光会发，还会看返回结果验证"。
7敏捷流程中测试人员参与哪些？▾
全程参与，不是等开发做完才进场：

· 需求评审：需求能不能测？有验收标准吗？（此时就要提）
· 站会：每天同步测试进度，有阻塞及时说。
· 开发中：写用例、备数据、跑冒烟测试。
· 评审会：展示测试结果，哪些过哪些没过。
· 回顾会：提改进——上次为什么漏测了。

核心理念：测试从需求阶段就介入，不是等开发完了测。
